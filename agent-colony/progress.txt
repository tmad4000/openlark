## Codebase Patterns

### Monorepo Structure
- Root: Turborepo with pnpm workspaces
- Apps: `apps/web` (Next.js 15), `apps/api` (Fastify)
- Packages: `packages/shared` (types, utilities)
- Each package has its own package.json, tsconfig.json, vitest.config.ts

### TypeScript Configuration
- Base config: `tsconfig.base.json` with strict mode
- Apps/packages extend base and set their own `outDir`, `rootDir`

### Testing
- Vitest for all packages
- API tests: Use `app.inject()` for HTTP testing without starting server
- Tests in `src/__tests__/` directory
- Use `passWithNoTests: true` in vitest.config for packages that may not have tests yet

### ESLint
- Root `eslint.config.js` uses `typescript-eslint` flat config
- Pattern: `tseslint.config()` wrapper with ignores first, then js.configs.recommended, then tseslint.configs.recommended

### Scripts
- `pnpm dev` / `pnpm build` / `pnpm test` / `pnpm lint` - via turbo
- `pnpm docker:up` / `pnpm docker:down` - Docker services
- `pnpm db:migrate` / `pnpm db:seed` - Database operations
- `SKIP_DB_TESTS=true pnpm test` - Skip integration tests requiring DB

### Module Structure (Iteration 2)
- Modules live in `apps/api/src/modules/<module-name>/`
- Each module has: `index.ts` (exports), `<name>.routes.ts`, `<name>.service.ts`, `<name>.schemas.ts`
- Routes use Fastify plugin pattern: `async function routes(app: FastifyInstance) { ... }`
- Schemas use Zod for validation
- Services contain business logic and database operations

### Auth Patterns (Iteration 2)
- JWT tokens in Authorization header: `Bearer <token>`
- Session stored in DB with token hash (not raw token)
- User password hashed with bcrypt (12 rounds)
- Sensitive fields (passwordHash, totpSecret) stripped from API responses

### Database Schema (Iteration 2)
- Drizzle ORM with PostgreSQL
- Schema files in `apps/api/src/db/schema/`
- All tables have: `id` (uuid), `createdAt`, `updatedAt`
- Multi-tenant: all user-facing tables have `org_id`
- Soft deletes: `deletedAt` timestamp, filter with `isNull(table.deletedAt)`

### Rate Limiting (Iteration 4, fixed Iteration 5)
- Use `@fastify/rate-limit` for brute force protection
- Pattern: Register in sub-scope to isolate from other routes
  ```ts
  await app.register(async (rateLimitedApp) => {
    await rateLimitedApp.register(rateLimit, {
      max: isTest ? 1000 : 10, // Higher limit in test mode
      timeWindow: "1 minute",
      errorResponseBuilder: (_request, context) => ({
        statusCode: 429,  // IMPORTANT: Must include for correct HTTP status
        error: "Too Many Requests",
        code: "RATE_LIMITED",
        message: `Too many requests. Please try again in ${context.after}.`,
      }),
    });
    rateLimitedApp.post("/login", handler);
  });
  ```
- Auth endpoints: 10 req/minute (production), 1000 (test)
- Return `{ statusCode: 429, code: "RATE_LIMITED", message: "..." }` on rate limit

### Drizzle Kit Compatibility (Iteration 5)
- Schema barrel file (`db/schema/index.ts`) must use `.ts` or no extension in imports
- drizzle-kit uses CJS loader which doesn't resolve `.js` extensions
  ```ts
  // CORRECT
  export * from "./auth";
  // WRONG (fails with drizzle-kit)
  export * from "./auth.js";
  ```

### Middleware Pattern (Iteration 4)
- Auth middleware in `modules/auth/middleware.ts`
- Export: `authenticate`, `requireAdmin`, `requirePrimaryAdmin`
- Use as `preHandler`: `{ preHandler: authenticate }`
- `req.user` available after authenticate (typed via module declaration)

### Integration Test Pattern (Iteration 10)
- Integration tests in `src/__tests__/<module>.integration.test.ts`
- Use `describe.skipIf(SKIP_DB_TESTS)` to skip when DB not available
- Clean up tables in `beforeEach` respecting foreign key order:
  ```ts
  // Messenger tables first (they reference auth tables)
  await db.delete(favorites);
  await db.delete(pins);
  await db.delete(messages);
  await db.delete(chatMembers);
  await db.delete(chats);
  // Then auth tables
  await db.delete(sessions);
  await db.delete(users);
  await db.delete(organizations);
  ```
- Helper functions for creating test users (register → get token/userId)
- vitest.config.ts must have `fileParallelism: false` and single fork pool
  to prevent database race conditions between test files

### WebSocket Pattern (Iteration 11, updated Iteration 12)
- Use `@fastify/websocket` plugin
- Endpoint at `/api/v1/messenger/ws?token=<jwt>` for authenticated connections
- Auth happens after WebSocket opens, close with code 4001 if invalid
- Separate Redis client for pub/sub subscriber (can't reuse publisher client)
- Lazy Redis connection initialization to avoid conflicts in tests
- Token payload uses `sub` not `userId` (match JWT convention)
- Event publishing: Call `publishMessageEvent(chatId, event)` after DB operations
- Notification: Call `notifyUserJoinedChat(chatId, userId)` when adding members - but ONLY if user is connected
- For test isolation: Call `resetWebSocketState()` in `beforeEach` to close connections and clear state
- Skip timing-sensitive pub/sub tests with `describe.skip()` for CI reliability

### Cross-Org Chat Pattern (Iteration 12)
- `getUserChats` returns all chats where user is a MEMBER, regardless of org
- This supports FR-2.29 (external groups with cross-org members)
- Chat membership is the security boundary, NOT org membership
- A user in org A can be a member of a chat created in org B
